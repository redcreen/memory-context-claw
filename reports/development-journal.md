## 2026-04-05 14:40 +0800

### 阶段主题

Phase 5 / live regression 工具链治理：给 `eval:hot*` 增加显式 hot-session 元数据，减少假失败排查成本。

### 触发原因

前面几轮已经确认：

- `smoke:eval` 是全绿的
- 但 `eval:hot*` 会因为 `main` 热会话没有真正隔离而出现假失败

问题不在于“我们知不知道这个 caveat”，而在于：

- 工具输出本身没有把这个风险直接暴露出来
- 每次看到失败都还要重新翻日志确认 `sessionKey`

### 这轮做了什么

1. 新增 hot-session 元数据提取模块

- [hot-session-regression.js](../src/hot-session-regression.js)

它会从 `openclaw agent --json` 返回里提取：

- `requestedSessionId`
- `observedSessionKey`
- `observedSessionId`
- `hotMainAlias`
- `matchedRequestedSessionId`
- `isolated`

2. 接回 `agent-regression.js`

现在每条 `eval:hot*` case 都会带上：

- `hotSession`

同时 summary 里新增：

- `isolated`
- `hotMainAlias`
- `matchedRequestedSessionId`

3. 接回治理周期报告

如果治理周期启用 `--run-live`，报告里也会显示：

- `isolated`
- `hotMainAlias`
- `matchedRequestedSessionId`

4. 同步更新文档

- [testsuite.md](../testsuite.md)
- [project-roadmap.md](../project-roadmap.md)
- [investigation-todo.md](investigation-todo.md)

### 为什么这样做

这一步不是修宿主，而是先把我们的工具链做诚实。

在宿主没有提供真正 clean live 入口之前，最重要的是：

- 不要再让 `eval:hot*` 像“普通回归”那样看起来干净
- 让每次失败都能立刻看出是不是会话污染

### 当前判断

这一步做完后，`eval:hot*` 的定位会更稳定：

- 它仍然不是 clean baseline
- 但它会是一个更可解释的 hot-session 健康检查

### 下一步

- 跑一次 `eval:hot:critical`，确认这些元数据字段能正确暴露
- 如果宿主仍没有隔离入口，就把这条限制继续固化进治理周期和维护基线

### 验证补充

我随后实际跑了：

```bash
npm run eval:hot:critical -- --timeout-ms 45000
```

结果非常清楚：

- `cases = 2`
- `passed = 1`
- `failed = 1`
- `isolated = 0`
- `hotMainAlias = 2`
- `matchedRequestedSessionId = 0`

并且两条 case 都显示：

- `observedSessionKey = agent:main:main`
- `observedSessionId = 103587ff-4d67-4598-8de7-ba361cf96fca`

这一步把结论彻底坐实了：

- `eval:hot*` 现在已经不会再“看起来像干净回归”
- 它会直接告诉我们：这轮其实跑在热 main 会话上

### 补充排查：`--local` 是否能绕开热会话

我这轮又试了两条本地嵌入路径：

```bash
openclaw agent --local --agent main --session-id codex-local-test-1 --message "只回复ok" --json --timeout 10
openclaw agent --local --to +8613900000000 --message "只回复ok" --json --timeout 10
```

观察结果：

- `--local --agent main --session-id ...` 直接命中了主会话锁
  - `session file locked (timeout 10000ms): pid=... /Users/redcreen/.openclaw/agents/main/sessions/103587ff-4d67-4598-8de7-ba361cf96fca.jsonl.lock`
- `--local --to ...` 没有产出可用 JSON，无法证明它提供了干净隔离会话

当前判断：

- 在这台机器上，`--local` 也不能被当成已经确认可用的 clean live 入口
- 所以这条线先正式收口成“宿主/工具链限制”
- 后面除非宿主版本或行为有变化，不再反复深挖这条入口

### 阶段主题

把“主体事实记忆”的真实 agent 回归收成一条轻量关键路径，不再每次都依赖完整 live suite。

### 触发原因

最近几轮我们已经把两个最关键的真实事实问题修通了：

- `我爱吃什么` 应该优先回到 `你爱吃牛排`
- `你怎么称呼我` 应该优先回到 `你叫刘超，我平时记你是超哥`

但现有的 `npm run eval:agent` 太重：

- 它会逐条调用真实 `openclaw agent`
- case 数越来越多
- 每次都全量跑，会把验证速度拖慢
- 对后续“每改一轮就回归一次”不友好

所以这轮要做的不是再改排序，而是把这两条主链路抽成一条更轻、更稳定的 live regression。

### 这轮做了什么

1. 给 `agent-regression.js` 增加了参数能力

- 支持 `--only case1,case2`
- 支持 `--cases /path/to/file.json`

这样脚本不再只能全量跑，而是可以按名字筛选一小组真实回归 case。

2. 增加了维护者脚本入口

在 `package.json` 里新增：

- `npm run eval:agent:critical`

当前固定只跑两条最关键的 live case：

- `food-preference-latest`
- `identity-preferred-name`

3. 补 README

把这条轻量路径写进了中英文 README，明确说明：

- `eval:agent` 仍然保留完整 live suite
- `eval:agent:critical` 是更适合频繁回归的关键事实验证路径

### 为什么这样做

这一步的目标不是“让测试更多”，而是让真实链路回归更像开发工具：

## 2026-04-06：查清 “我爱吃牛排” 为什么没被召回

### 这轮为什么做

用户在 `main` agent 中说了“我爱吃牛排”，随后发生了 `compact` 和 `/new`。  
新会话里再问“我爱吃什么”，结果答成了旧的“爱吃面食”。

这个问题涉及两条非常关键的链路：

- 候选记忆链路为什么没有把“牛排”接住
- OpenClaw 内置 `memory_search` 为什么没有把一个已经写入并索引的记忆召回来

### 这轮查到了什么

1. 不是时序问题

- `food-preference` 文件已经在查询前写出：
  - `/Users/redcreen/.openclaw/workspace/memory/2026-04-05-food-preference.md`
- 它也已经在查询前进入 `main.sqlite`
- 所以不是 `/new` 太快、索引还没完成
- 更具体的时间线是：
  - `09:29:21` 左右，OpenClaw session-memory hook 写出 `2026-04-05-food-preference.md`
  - `09:29:31` 左右，该文件进入 `main.sqlite`
  - `09:30` 之后，你在新会话里问“我爱吃什么”
- 所以这是“已写入 + 已索引，但没被召回”的问题，不是索引 race condition

2. 不是 source filter 把 memory 挡掉了

- `main.sqlite` meta 明确显示：
  - `sources = ["memory", "sessions"]`
- OpenClaw 的 `buildSourceFilter()` 只是把 source 拼成 `IN (?, ?)`
- 所以这次不是 `memory` 源被过滤掉

3. 参数微调也救不回来

- 做了多组 `memorySearch.query.hybrid.*` 实验：
  - temporalDecay
  - 更高 textWeight
  - 更大 candidateMultiplier
  - MMR
- 报告在：
  - `/Users/redcreen/Project/长记忆/context-assembly-claw/reports/memory-search-param-eval.md`
- 结果一致：
- `top 10 / top 20 / top 30` 都没有把 `food-preference.md` 拉进候选池
- 同时我还验证了更直接的查询：
  - `openclaw memory search --query '牛排'`
  - `openclaw memory search --query '我爱吃牛排'`
  - `openclaw memory search --query '牛排 刘超'`
- 结果仍然没有把 `memory/2026-04-05-food-preference.md` 打进候选池

4. temporalDecay 实际上对这条文件没有作用

- OpenClaw 当前只会从两类路径提取日期：
  - `memory/YYYY-MM-DD.md`
  - evergreen memory（如 `MEMORY.md`）
- `memory/2026-04-05-food-preference.md` 不匹配日期文件正则
- 所以 temporalDecay 对这条文件实际上不起作用

5. 关键词层对这类中文短偏好句基本没帮上忙

- `main.sqlite` 记录的 FTS tokenizer 是：
  - `unicode61`
- 实测：
  - `chunks_fts MATCH '牛排'` 返回 `0`
  - `chunks_fts MATCH '牛排 AND 刘超'` 返回 `0`
  - `chunks_fts MATCH '爱吃'` 只会打到旧的 `sessions/...`
- 当前 FTS 查询构造还是把 token 用 `AND` 全连起来
- 这对中文短偏好句和混合检索词并不友好
- 这次最直观的证据是：
  - `chunks` 表里用 `instr(text, '牛排')` 能明确查到
  - 但 `chunks_fts MATCH '牛排'` 返回 `0`
- 也就是说，原文在 chunk 里，但 FTS 层没有把它变成可用的中文短词命中

6. 文件形态本身也很不利于向量召回

- `food-preference.md` 当前只被切成了 `1` 个 chunk
- 这一个 chunk 的前面大段都是：
  - session startup instruction
  - metadata
  - greeting
- “我爱吃牛排”虽然在 chunk 里，但位置很靠后
- 这意味着 embedding 更像是在表达：
  - “一次 session 启动摘要”
  - 而不是一个干净、强语义的“用户饮食偏好”片段
- 我直接看了文件内容，前半部分主要是：
  - Session 时间
  - Session Key / ID / Source
  - Session Startup 指令
  - greeting
  - metadata
- 最后才出现：
  - `我爱吃牛排`
  - `记住了，你爱吃牛排`
- 这意味着它当前更像一个“带记忆内容的会话记录”，不是“可检索的记忆卡片”

7. 旧的 sessions 语料在这个查询上形成了明显覆盖

- `memory_search` 实际返回的 top 结果几乎全是旧的 `sessions/...`
- 内容集中在：
  - “爱吃面食”
  - “面条、馒头”
  - “因控体重和血压暂时少吃”
- 这些旧语料数量更多、表述更集中、历史上被反复提过
- 所以新的“牛排”在当前文件形态下，很难和这些旧 session chunk 竞争

### 当前根因判断

这次不是单点故障，而是 4 个因素叠加：

1. `food-preference.md` 是 session summary 形态，不是为检索优化的记忆片段
2. 它只形成了一个大 chunk，关键信息埋在后面
3. `unicode61` 下中文短词 FTS 很弱，关键词层帮不上忙
4. temporalDecay 对这种文件命名不起作用

结果就是：

> 文件写了、索引了，但既没有在关键词层被命中，也没有在向量层赢过旧的“爱吃面食”历史 session。

### 对插件自己的影响

我们的 candidate memory 回捞当前只读取：

- `source = 'sessions'`
- `path LIKE 'sessions/%'`

它不会消费 OpenClaw 旁路写出的 `memory/*.md` session-memory 文件。  
所以即使宿主已经把“牛排”写进 `memory/2026-04-05-food-preference.md`，我们的候选记忆报告依然看不到它。

### 这轮结论

这次最重要的不是“再调一个参数”，而是把边界看清了：

- 这不是索引没完成
- 也不是简单 rerank 能救
- 而是当前 session-memory 文件形态和宿主检索机制不适合这类“短而明确的用户偏好记忆”
- 更准确地说，这是三层问题叠加：
  1. 宿主 session-memory 写出的文件，不是为检索优化的记忆单元
  2. 默认 FTS/tokenizer 对中文短词帮助不大
  3. 我们自己的候选记忆链路又没有消费这类 `memory/*.md`

### 下一步

1. 插件侧：
   - 让候选记忆链路开始消费 `memory/*.md` 的 session-memory 文件
2. 检索侧：
   - 研究更适合检索的 session-memory 文件形态或更细 chunking

## 2026-04-05：把“牛排”从策略句收敛成用户事实句

2. 但 `buildCandidate('你爱吃牛排', ...)` 一开始返回 `null`

- “短句过滤”前半段已经认识到它是高价值 anchored fact
- 但后半段忘了这个例外
- 于是把真正该保留的用户事实又过滤掉了

  - `默认把...`

这样做不是为了硬编码“牛排”，而是把规则抽象成：

- 助手根据事实延伸出来的执行策略退场
