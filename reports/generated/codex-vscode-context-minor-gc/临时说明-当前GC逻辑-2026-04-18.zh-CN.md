这份文档只解释当前这个仓库里，`VS Code Codex + UMC Context Minor GC` 现在到底怎么工作，以及你该看哪个文件。

当前已经生效，不再是只做 shadow 记录。

- `applied=true`
- `relation=continue`
- `baseline_context_estimate=1459`

## 现在的真实逻辑

当前有两层压缩：

### 第一层：Codex 宿主原生压缩

这是 VS Code/Codex 自己做的，负责长对话时的宿主级 compaction。

### 第二层：这个项目的 UMC Context Minor GC

这是我们自己做的 repo 级 working-set 裁剪，负责把“当前任务不需要的旧 turn”剔掉，只保留主线内容。

本次改动后，这一层的默认规则已经放宽：

- 以前：只有 `switch` / `resolve` 才真正 apply
- 现在：`continue` 也允许真正 apply

所以你现在这种“连续追问”的对话，也会真正裁剪，而不是只记日志。

## 为什么你之前会感觉“看起来开了，但没减少”

因为之前是这个状态：

- 决策已经跑了
- 理论上有 reduction
- 但 `relation=continue`
- 默认 guard 不放行
- 所以最终 `effective_context_estimate` 还是 baseline

也就是说，之前很多轮是“看到了潜在收益”，但“没有真的替换进最终 prompt”。

现在这点已经改了。

## 你以后怎么判断有没有生效

只看这一条就够了：

- `codex-telemetry.jsonl` 里最新一行是否为 `applied=true`

然后再看这两个数：

- `baseline_context_estimate`
- `effective_context_estimate`

如果后者更小，就说明这轮真的裁剪了。

## 以后每次回复怎么看

现在会尽量在回复最后附一行 footer，格式类似：

`Context(估算): 本轮发送context 151k，宿主context 152k；窗口总大小 258k，压缩比：0.6%（151k/152k），占总窗口比例：58.4%（151k/258k）`

解释：

- `宿主层 a`
  当前这轮实际发送给 Codex 宿主的大盘子 context 估算
- `宿主层 o`
  如果没有项目层 GC，本轮原始逻辑大致会发送多少
- `项目层 a`
  当前 UMC 真正保留下来的 context 大小
- `项目层 o`
  没做 UMC GC 时，这个项目层原本会放进去多少

一句话：

- 宿主层看“总体”
- 项目层看“UMC 自己到底压了多少”
- footer 主要看“这轮实际发了多少、原本会发多少、占窗口多少”
