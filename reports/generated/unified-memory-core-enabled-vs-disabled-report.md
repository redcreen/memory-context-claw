# Unified Memory Core 开启 / 关闭 对比测试报告

## 结论
开启 `unified-memory-core` 后，OpenClaw 在这组测试里会稳定拿到额外的长期记忆上下文；关闭后，这一层上下文组装完全消失。

一句话说：

- **开启插件**：每个问题平均会多出 `3.5` 个被选中的记忆片段，平均多出 `2534` 个字符的上下文补充。
- **关闭插件**：被选中的记忆片段是 `0`，上下文补充字符数也是 `0`。

## 测试方式
使用脚本：

```bash
cd /Users/redcreen/Project/长记忆/unified-memory-core
npm run eval:toggle
```

这组对比不是改动主模型，而是只比较 `unified-memory-core` 这层 **context assembly** 是否工作：

- 开启：使用插件默认 `safe-local` 配置
- 关闭：保持同样的问题输入，但把插件 `enabled=false`

## 总体结果

- 测试问题数：`8`
- 开启插件时，有记忆片段进入上下文的问题数：`8`
- 关闭插件时，有记忆片段进入上下文的问题数：`0`
- 开启插件时，平均选中片段数：`3.5`
- 开启插件时，平均上下文补充长度：`2534.375` 字符
- 关闭插件时，平均选中片段数：`0`
- 关闭插件时，平均上下文补充长度：`0`

## 代表性差异

### 1. 概念解释类
问题：

`我之前怎么理解 Lossless 和长期记忆的区别？`

开启插件时，选中了：

- `MEMORY.md`
- `openclaw-memory-vs-lossless.md`
- `memory/2026-04-04.md`
- `memory/2026-04-02.md`

关闭插件时：

- 没有任何记忆片段进入上下文

这说明开启后，模型会更容易拿到“长期规则 + 专题说明 + 近期过程”的组合背景。

### 2. 配置说明类
问题：

`unified-memory-core 这个插件的配置应该怎么写？`

开启插件时，选中了：

- `unified-memory-core-config.md`
- `unified-memory-core-roadmap.md`

关闭插件时：

- 没有任何插件配置相关上下文

这说明开启后，模型更容易直接回答“这个插件本身怎么配置”，而不是只给泛 OpenClaw 配置。

### 3. 换一种说法的召回
问题：

`为什么已经有长记忆了，还要这个上下文插件？`

开启插件时，选中了：

- `MEMORY.md`
- `openclaw-memory-vs-lossless.md`
- `unified-memory-core-config.md`
- `unified-memory-core-roadmap.md`

关闭插件时：

- 没有任何记忆片段进入上下文

这说明查询改写 + 重排链路对“长记忆 / 上下文插件”这类换说法已经是有效的。

## 我们现在能明确说的区别

### 开启插件时
- 会主动从长期记忆库里召回相关片段
- 会做路径过滤、查询改写、规则重排
- 会把选中的内容装进 `systemPromptAddition`
- 对“概念解释、目录规则、插件配置、换说法召回”都有明显帮助

### 关闭插件时
- 不会进行这层长期记忆上下文组装
- 不会有额外记忆片段进入当前 prompt
- 回答更依赖模型已有知识和当前会话残留

## 当前边界
这份对比主要证明的是：

- **插件有没有把长期记忆真正送进当前上下文**

它还不是“最终回答质量”的全部评测，因为最终回答还会受：

- 主模型本身能力
- 当前会话历史
- OpenClaw 其他默认注入内容

影响。

但这份测试已经足够说明：

**`unified-memory-core` 打开的核心价值，不是另一个记忆库，而是把长期记忆稳定地转成当前轮真正可用的上下文。**
