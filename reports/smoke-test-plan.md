# Smoke 测试计划

## 为什么现在就要整理

现在整理 smoke 案例是对的，而且已经是必要动作。

原因不是“测试越多越好”，而是当前系统已经进入这些阶段：

1. `cardArtifact` 已经开始在真实 OpenClaw 回答里生效
2. 主体事实类型已经从：
   - 偏好
   - 身份
   - 规则
   扩到：
   - 个人背景
   - 项目定位
3. 系统已经开始处理“新旧事实冲突”

如果这时还没有一套明确的 smoke case，很容易出现：

- 功能一直加
- 但不知道哪些核心链路必须长期稳定
- 后面再补案例时，已经很难回忆“哪一类问题最关键”

## 现在这套 smoke 应该覆盖什么

我建议把 smoke 分成 4 层，而不是混成一堆问题。

### 1. 用户主体事实

目标：

- 新事实是否能被优先使用
- 旧冲突事实是否被压下去

当前必测：

- `我爱吃什么`
- `你怎么称呼我`
- `我生日是什么时候`
- `我女儿叫什么，生日是哪天，现在几年级`

后续可扩：

- `你有几个孩子`
- `你现在做什么行业`

### 2. 长期规则

目标：

- `MEMORY.md` / 目录规则 / 工作方式 是否能优先回到卡片化结论

当前必测：

- `MEMORY.md 应该放什么内容`
- `长期记忆目录规则是什么`

### 3. 概念理解

目标：

- 记忆层、上下文层、provider 这类核心概念，是否还能稳定回到对的资料

当前必测：

- `为什么已经有长期记忆了，还需要 Lossless`

### 4. 项目事实

目标：

- 插件本身的定位、配置、项目长期背景 是否能稳定回到对的材料

当前必测：

- `unified-memory-core 这个插件的配置应该怎么写`
- `这个项目主要解决什么问题`

## 当前建议分层

### A. 轻量 smoke

用途：

- 每轮开发都跑
- 快速看 retrieval / ranking / assembly 有没有明显退化

当前入口：

```bash
npm run smoke:eval:critical
```

当前只覆盖两层：

- `user-facts`
- `rules`

这些是“每轮必跑”的最低保护面。

### B. 全量 smoke baseline

用途：

- 阶段性看当前覆盖面是否够
- 提前暴露还没收稳的意图类型

当前入口：

```bash
npm run smoke:eval
```

当前 smoke case 文件：

- [smoke-cases.json](../evals/smoke-cases.json)

### C. 轻量真实 agent 回归

用途：

- 验证真实 `main` agent 的最关键事实是否还活着

当前入口：

```bash
npm run eval:agent:critical
```

当前只盯：

- `我爱吃什么？`
- `你怎么称呼我？`

### D. 完整真实 agent 回归

用途：

- 发布前或阶段性确认

当前入口：

```bash
npm run eval:agent
```

## 这套 smoke 现在够不够

当前判断：

- 对“偏好 / 规则”已经足够进入每轮 smoke
- 对“身份 / 概念 / 配置”已经开始覆盖，但还没全部收稳
- 对“个人背景 / 项目定位”开始覆盖，但还不够强
- 对“冲突治理”还应该再补一两条更明确的 case

刚跑过一次全量 baseline，结果是：

- `7` 条 case
- `5` 条通过
- `2` 条失败

当前失败项：

- `identity-name`

后续说明：

- `memorySearch.provider 是做什么的` 更依赖最终回答层，不适合继续作为 assemble-only smoke 的硬门槛，更适合留在 live regression / watchlist。

这正说明把 smoke 案例提前整理出来是值得的，因为它已经开始帮我们暴露“还没收稳”的类型。

## 下一步建议补的 smoke case

### 优先补

- `这个项目主要解决什么问题`
- `你现在做什么行业`
- `你有几个孩子`

### 第二批再补

- `我还爱吃哪些`
- `我的长期工作方式是什么`
- `这个插件和 OpenClaw 原生 memory 的关系是什么`

## 一句话结论

`是的，现在整理 smoke 案例正是时候；而且应该先按“用户事实 / 长期规则 / 概念理解 / 项目事实”四层收起来。`
